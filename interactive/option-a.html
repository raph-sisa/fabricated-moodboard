Introduction to Claude Code

Claude Code is an agentic coding tool that lives in your terminal. Unlike traditional code completion, Claude Code understands your entire project context вҖ” files, dependencies, build systems вҖ” and can make multi-file changes, run commands, and iterate on errors autonomously.

In this workshop, we will use Claude Code as a co-pilot specifically for hardware projects: generating Arduino sketches, debugging sensor readings, writing communication protocols, and designing PCB layouts.

Why Claude Code for Hardware?

Embedded development has unique challenges: limited memory, real-time constraints, hardware-specific APIs, and the gap between documentation and reality. Claude Code bridges this gap by understanding datasheets, suggesting pin configurations, and generating tested firmware patterns.

// Ask Claude Code to generate sensor reading code const reading = await sensor.read({ pin: 'A0', type: 'analog', samples: 10, interval: 50 }); // Claude suggests averaging for noise reduction const averaged = reading.reduce((a, b) => a + b) / reading.length;

By the end of this module, you will be comfortable using Claude Code to accelerate every phase of your hardware project вҖ” from ideation through deployment.

Setup & Configuration

Before diving in, let's get your development environment configured. Claude Code runs in your terminal alongside your existing toolchain.

Installation

# Install Claude Code via npm npm install -g @anthropic-ai/claude-code # Verify installation claude --version # Authenticate claude auth login

Project Configuration

Create a .claude directory in your project root to store context and preferences. This helps Claude understand your hardware setup вҖ” which board you are using, pin assignments, and library preferences.

# Initialize Claude Code in your project cd ~/my-arduino-project claude init # This creates .claude/ with default config # Edit .claude/settings.json for hardware specifics

The settings file lets you specify your target board, serial port, and commonly used libraries so Claude can generate compatible code from the start.

Prompting for Hardware

Writing effective prompts for hardware projects requires a different approach than web development. You need to specify physical constraints, timing requirements, and electrical characteristics.

The Hardware Prompt Template

A good hardware prompt includes: the board, the component, the communication protocol, and the desired behavior. Be specific about pin numbers and voltage levels.

// Example prompt structure: // "Using Arduino Nano with BME280 on I2C (SDA=A4, SCL=A5), // read temperature every 2 seconds, display on // SSD1306 OLED (same I2C bus, address 0x3C), // and log to SD card on SPI (CS=10)"

The more specific you are about your physical setup, the more accurate Claude's generated code will be. Always mention power constraints if running on battery.

Firmware Generation

This is where Claude Code truly shines. Rather than piecing together library examples, you describe the behavior you want and Claude generates production-ready firmware.

From Description to Working Code

// Claude generates complete, compilable sketches #include <Wire.h> #include <Adafruit_BME280.h> #include <Adafruit_SSD1306.h> Adafruit_BME280 bme; Adafruit_SSD1306 display(128, 64, &Wire); void setup() { Wire.begin(); bme.begin(0x76); display.begin(SSD1306_SWITCHCAPVCC, 0x3C); }

Claude handles library includes, initialization sequences, error checking, and even suggests pull-up resistor values for I2C lines.

Debugging with Claude

When your hardware does not behave as expected вҖ” and it won't вҖ” Claude Code can help diagnose issues from serial output, timing diagrams, and error codes.

Common Debugging Patterns

Share your serial monitor output with Claude. It can identify initialization failures, I2C address conflicts, timing issues, and memory overflows that are common in embedded development.

// Claude helps add diagnostic output void diagnoseI2C() { for (byte addr = 1; addr < 127; addr++) { Wire.beginTransmission(addr); if (Wire.endTransmission() == 0) { Serial.print("Found: 0x"); Serial.println(addr, HEX); } } }

Advanced Patterns

Once you are comfortable with the basics, Claude Code can help you implement more sophisticated patterns: state machines, interrupt-driven architectures, low-power modes, and OTA updates.

State Machine Pattern

enum State { IDLE, READING, DISPLAYING, LOGGING, SLEEPING }; State currentState = IDLE; void loop() { switch (currentState) { case IDLE: if (millis() - lastRead > INTERVAL) currentState = READING; break; case READING: readSensors(); currentState = DISPLAYING; break; // ... Claude generates the full pattern } }

These patterns make your firmware robust, maintainable, and power-efficient вҖ” exactly what you need for real-world hardware deployments.

What is Arduino?

Arduino is an open-source electronics platform based on easy-to-use hardware and software. At its core, an Arduino board contains a microcontroller вҖ” a small computer on a single chip вҖ” that you can program to read inputs (a sensor value, a button press) and turn them into outputs (activating a motor, turning on an LED, publishing data online).

In this workshop, we will primarily use the Arduino Nano вҖ” a compact, breadboard-friendly board powered by the ATmega328P. It has 14 digital I/O pins, 8 analog inputs, and communicates over USB serial.

The Arduino Ecosystem

Arduino is more than a board вҖ” it is a complete ecosystem: the IDE for writing and uploading code, a massive library collection, a global community, and a philosophy that hardware should be accessible to everyone. Every sketch (Arduino program) has two core functions:

void setup() { // Runs once when the board powers on // Initialize pins, serial, sensors here } void loop() { // Runs repeatedly forever // Your main program logic goes here }

This structure вҖ” initialize, then loop вҖ” maps naturally to how physical devices work: start up, then continuously respond to the world.

Blink: Your First Sketch

Every Arduino journey starts with Blink. It is the "Hello, World" of physical computing вҖ” the simplest proof that your board, your code, and your connection are all working.

The Blink Sketch

// Blink вҖ” the hardware "Hello, World" // Toggles the built-in LED every second void setup() { pinMode(LED_BUILTIN, OUTPUT); } void loop() { digitalWrite(LED_BUILTIN, HIGH); // LED on delay(1000); // Wait 1 second digitalWrite(LED_BUILTIN, LOW); // LED off delay(1000); // Wait 1 second }

Breaking it Down

pinMode(LED_BUILTIN, OUTPUT) tells the microcontroller that pin 13 (the built-in LED) should be used as an output вҖ” meaning we will send voltage out through it, not read voltage from it.

digitalWrite(LED_BUILTIN, HIGH) sets the pin to 5V, which turns the LED on. LOW sets it to 0V, turning it off. The delay(1000) pauses execution for 1000 milliseconds.

Upload this sketch, and you will see the small LED on your Nano blinking steadily. You have just programmed a physical object. Everything else builds from here.

Digital I/O

Digital pins deal in absolutes: HIGH (5V) or LOW (0V). They are perfect for buttons, switches, LEDs, and any binary signal. The Arduino Nano has 14 digital pins (D0вҖ“D13).

Reading a Button

const int BUTTON_PIN = 2; const int LED_PIN = 13; void setup() { pinMode(BUTTON_PIN, INPUT_PULLUP); pinMode(LED_PIN, OUTPUT); } void loop() { int state = digitalRead(BUTTON_PIN); digitalWrite(LED_PIN, !state); }

INPUT_PULLUP activates an internal pull-up resistor, so the pin reads HIGH when the button is not pressed and LOW when it is. This eliminates the need for an external resistor.

Analog Input

While digital pins see the world in black and white, analog pins see shades of grey. The Arduino's ADC (Analog-to-Digital Converter) reads voltages between 0V and 5V and maps them to values between 0 and 1023.

Reading a Potentiometer

const int POT_PIN = A0; void setup() { Serial.begin(9600); } void loop() { int value = analogRead(POT_PIN); float voltage = value * (5.0 / 1023.0); Serial.print("Voltage: "); Serial.println(voltage); delay(100); }

This is the foundation of sensor reading. Temperature sensors, light sensors, flex sensors вҖ” they all produce analog voltages that we convert to meaningful numbers.

Serial Communication

Serial is your window into the microcontroller's mind. It lets your Arduino send and receive data over USB, which is invaluable for debugging and data logging.

void setup() { Serial.begin(115200); // Fast baud rate Serial.println("System initialized"); } void loop() { if (Serial.available()) { char cmd = Serial.read(); processCommand(cmd); } }

The Serial Monitor in the Arduino IDE (or any terminal emulator) displays this output in real-time. You can also send commands back to the Arduino.

Libraries & Sensors

Arduino's power comes from its library ecosystem. Instead of writing low-level I2C commands to talk to a temperature sensor, you install a library and call bme.readTemperature().

Using the BME280

#include <Adafruit_BME280.h> Adafruit_BME280 bme; void setup() { Serial.begin(9600); if (!bme.begin(0x76)) { Serial.println("BME280 not found!"); while(1); } } void loop() { Serial.print(bme.readTemperature()); Serial.println(" C"); delay(2000); }

I2C & SPI

These are the two most common communication protocols in the embedded world. I2C uses two wires (SDA, SCL) and supports multiple devices on a single bus. SPI uses four wires but is faster, making it ideal for displays and SD cards.

// I2C Scanner вҖ” find all devices on the bus #include <Wire.h> void setup() { Wire.begin(); Serial.begin(9600); for (byte addr = 1; addr < 127; addr++) { Wire.beginTransmission(addr); if (Wire.endTransmission() == 0) { Serial.print("Device at 0x"); Serial.println(addr, HEX); } } }

Project Integration

Now we bring it all together. A real project combines multiple sensors, displays, storage, and communication вҖ” all running concurrently on a tiny microcontroller.

Architecture Pattern

Use a task-based approach instead of blocking delays. This lets your firmware handle multiple responsibilities without missing sensor readings or user inputs.

unsigned long lastSensorRead = 0; unsigned long lastDisplayUpdate = 0; void loop() { unsigned long now = millis(); if (now - lastSensorRead > 2000) { readSensors(); lastSensorRead = now; } if (now - lastDisplayUpdate > 500) { updateDisplay(); lastDisplayUpdate = now; } }

This non-blocking pattern is essential for the projects you will build in the workshop. Your Weather Station, for example, needs to read sensors, update the display, and log to the SD card вҖ” all appearing to happen simultaneously.
